# Шаблон ручной проверки базисных точек

Статус:
частично ручной workflow.

## Что содержит этот файл

- Уже внесённые ручные правки сохраняются между запусками пайплайна.
- Полуавтоматические candidate-строки могут добавляться после UTF-16 string extraction из legacy `.doc`.
- Каждая candidate-строка должна быть вручную сверена с исходным документом до научного использования.

## Как заполнять колонки

- `base_point_name`: сохраняйте исходное написание обозначения точки.
- `obs_date`: используйте ISO-дату только если в источнике есть полная явная дата.
- `x_m`, `y_m`: переносите координаты ровно как указано; не меняйте единицы и CRS молча.
- `accuracy_m`: заполняйте только если точность измерения прямо указана в источнике.
- `point_status`: допустимые нормализованные значения — `new`, `reinstalled`, `calculated`, `refined`, `original`, `unknown`.
- `status_note`: сохраняйте исходную формулировку, которая обосновывает статус.
- `site_id`: заполняйте вручную, если он не был определён автоматически.
- `qc_flag`: сохраняйте parser-флаги и при необходимости добавляйте свои прозрачные теги, например `MANUAL_VERIFIED`, когда проверка завершена.
- `qc_note`: объясняйте, почему `site_id`, статус или дата остаются неопределёнными.
- `source_row_ref`: сохраняйте стабильную ссылку вроде `strings_el:115` или ручную пометку вида `page 2, paragraph 3`.

## Правила нормализации статуса

- `Новый` -> `new`
- `Переустановлен` -> `reinstalled`
- `Расчетные координаты` / `Расчётные координаты` -> `calculated`
- `Уточнены` / `Уточнение координат` -> `refined`
- иначе -> `original` или `unknown`

## Важно

- Не придумывайте координаты и даты.
- Оставляйте сомнительные значения пустыми и объясняйте неопределённость в `qc_note` или `status_note`.
- Строки с `SITE_ID_UNRESOLVED` считаются ожидаемыми до тех пор, пока сопоставление точки с участком не подтверждено вручную.
